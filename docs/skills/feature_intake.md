# Feature Intake Workflow

## Purpose
- Define the repeatable intake workflow for new features and major feature expansions.
- Keep feature requests in premise-definition mode until product, UX, and market-data boundaries are explicit enough to build.

## Core Intake Checks
- trader interruption cost
- primary surface
- glance time and readability
- user-visible states
- data source
- refresh policy
- battery impact
- failure and retry states
- future-scope boundaries
- validation path

## Output
- product premises
- open decisions
- safe autonomous assumptions
- validation path
- whether Product, UX/UI, Market Data, Engineer, QA, or Historian must participate

## Rule
- Ask only the smallest set of high-leverage numbered questions needed to define the accepted scope.
